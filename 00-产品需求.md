# Super Guild (The Shelter Program) - MVP 产品需求文档大纲

## 1. 项目愿景与目标 (Vision & Objectives)

- **1.1 产品定义**：面向 AI 时代超级个体的去中心化协作协议与基础设施（公会制底座）。
    
- **1.2 核心愿景**：解构传统公司制，为超级个体提供一个无需许可、中立且高效的“数字公会”，实现生产力的彻底解放。
    
- **1.3 核心竞争力 (四大支柱)**：
    
    - **万能中后台**：货架化服务，一站式解决非创造性后勤琐事。
        
    - **去中心化 DAO**：基于代码和 NFT 的治理体系，消除对中心化平台的信任依赖。
        
    - **零费率协议**：本金 100% 结算，通过协议收益（如未来托管生息）覆盖支出。
        
    - **AI 原生接口**：预留 AG-UI 标准接口，未来可无缝接入 AI 智能体助理。
        

## 2. 用户角色定义 (Target Users)

- **2.1 统一身份**：**超级个体 (Super Individual)**。
    
    - 系统不强制区分“雇主”与“雇员”，每个成员皆可发布任务（发起协作）或承接任务（贡献价值）。
        
- **2.2 用户旅程 (User Path)**：
    
    - 连接钱包 -> 获得初始 VCP -> 浏览中后台货架 -> 发起/参与协作 -> 提交凭证 -> 获得结算与声誉 -> 晋升并获取 NFT 勋章。
        

## 3. 核心功能模块：身份与声誉 (Identity & VCP)

- **3.1 极简准入**：支持 Web3 钱包（Wagmi v3）一键登录。
    
- **3.2 创世空投**：新用户首次绑定钱包，系统自动发放 **100 VCP** 初始值（存入 Supabase 镜像库）。
    
- **3.3 声誉记录**：任务结算后，根据任务金额与评价自动增发 VCP。MVP 阶段不涉及外部数据（GitHub/LinkedIn）导入。
    

## 4. 核心功能模块：货架化万能中后台 (Universal Back-office)

### 4.1 核心设计逻辑：全员平权的生产力货架

万能中后台是公会为超级个体提供的“补给站”。为了践行公会制取代公司制的愿景，这一部分的功能设计遵循以下原则：

- **平权原则**：对所有钱包持有者一视同仁。VCP 仅在后台记录信用，不作为中后台服务的准入门槛，不影响服务定价。
    
- **货架模式**：服务以“商品”形式陈列，采用 SKU（库存单位）管理，支持运营方随时增删改。
    
- **支付机制**：全线采用主流加密稳定币（USDT/USDC 等）支付，确保公会财务透明及外部服务商（如法律、财税专家）的即时结算。
    

### 4.2 频道一：超级个体基础设施 (Base Infrastructure)

**定位**：提供超级个体独立生存的“标准化操作指南”，通过买断制解锁沉淀知识。

- **计费模式**：**类目买断制 (Category Buyout)**。
    
- **服务菜单 (SKUs)**：
    
    1. **【基础作业规范】**：包含分布式协作的文档流、代码规范、个人项目管理 SOP 等。
        
    2. **【数字游民安全手册】**：包含跨境资产保护、网络环境搭建、隐私脱敏作业指南。
        
    3. **【AI 创业合规指引】**：包含各主要法区对 AI 生成内容的版权声明、算法备案流程等。
        
- **交互逻辑**：支付后，该类目在用户 Dashboard 中永久解锁。用户可随时在线查阅、调用模版或接收版本更新通知。
    

### 4.3 频道二：可插拔专项服务 (Pluggable Solutions)

**定位**：针对具体场景的专项解决方案，提供“拆箱即用”的工具包或文档。

- **计费模式**：**方案零售制 (Pay-per-Item)**。每个具体方案独立定价，精准付费。
    
- **服务菜单 (SKUs)**：
    
    - **【法律合规】**：合伙协议模版、数据处理协议、知识产权保护声明。
        
    - **【财税统筹】**：跨境结算税务合规工具、个人工作室税务规划方案。
        
    - **【IP 防御】**：品牌全球查重工具、链上存证维权路径。
        
    - **【跨境模型】**：多币种结算风险规避方案、跨境劳动合同适配。
        
    - **【自媒体运维】**：多平台内容分发策略、创作者权益保护模版。
        
- **交互逻辑**：用户按需点击具体方案，支付后获得文件下载权限、API 访问 Key 或对应的数字工具访问权。
    

### 4.4 频道三：专家专项咨询服务 (Expert Consultation)

**定位**：连接经过公会认证的各领域顶级专业人士，解决个性化的高阶问题。

- **计费模式**：**预约付费制 (Appointment Booking)**。
    
- **服务菜单**：
    
    - **专业人士名片池**：展示专家头像、领域标签（如：Web3 财税专家、海外合规律师）、过往成功案例数。
        
    - **预约功能**：选择时间窗口 -> 支付定金/全额 -> 进入专属协作室。
        
- **交互逻辑**：公会作为信任背书方，在服务完成后根据双方确认解锁资金。
    

### 4.5 开发架构细化：开放式货架协议 (Open Shelf Protocol)

为了实现“随时增加服务品类”，开发团队需在 **Supabase** 中实现一套动态渲染引擎。

#### 1. 数据库结构（关键表：`Service_Store`）

开发需建立以下字段，以确保你只需在后台填入数据，前端即可自动上架：

- `channel_id`: 频道 ID (1:基础, 2:专项, 3:专家)
    
- `category_name`: 栏目名称 (如：法律合规)
    
- `item_name`: 方案名称 (如：合伙协议模版)
    
- `price`: 价格 (以数字表示)
    
- `currency_type`: 币种 (默认为 USDT)
    
- `payment_type`: 支付类型 (CATEGORY_BUYOUT / ITEM_PAY)
    
- `content_ref`: 内容引用（Markdown 地址、文件下载路径或专家 Profile ID）
    

#### 2. 前端展示逻辑 (Marketplace Rendering)

    - 已购买：蒙层消失，变为“进入/下载”按钮。

### 4.6 总结：VCP 与中后台的关系

在 Super Guild 的体系中，**中后台是“服务于人”的，而 VCP 是“衡量于人”的**：

- 用户在中后台的每一笔购买和成功的协作，虽然不需要 VCP 启动，但在任务结项后，会产生新的 VCP 记录，反哺其信用体系。
    
- **公会信条**：我们可以通过 VCP 决定谁来管理公会（治理），但公会必须向所有成员提供同等的生存支持（中后台）。

## 5. 任务协作与进度管理系统 (Task Collaboration & Progress)

### 5.1 核心定义：公会成员间的网状协作

任务协作系统是公会成员实现价值交换的场所。它不依赖于任何中心化管理，而是通过“协议”来保障协作的公平。

- **协作逻辑**：成员自发发布任务，自由组合，基于里程碑进行分阶段结算。
    
- **业务边界**：该模块仅处理成员 A 与成员 B 之间的任务逻辑，与公会提供的中后台服务完全独立。
    

### 5.2 任务全生命周期管理 (Task Lifecycle)

系统通过“状态机”管理每一个协作请求，确保进度透明。

1. **发布态 (Open)**：成员发起协作，定义里程碑、预算与交付要求。
    
2. **锁定态 (Locked)**：发起人确认资金托管，此时任务正式由“意向”转为“合同”。
    
3. **执行态 (Active)**：协作双方开始交付，系统根据里程碑进行进度追踪。
    
4. **验收态 (Pending Approval)**：执行方上传**任务凭证**，等待发起方确认。
    
5. **结算态 (Settled)**：验收通过，资金释放，信誉（VCP）同步增发。
    
6. **争议态 (Disputed)**：若验收不通过或产生违约，进入公会仲裁流程。
    

### 5.3 凭证解锁制与证据链规范 (Proof-of-Work)

这是公会“无需自我证明”的核心。每一个里程碑的解锁必须基于**真实证据**，而非口头承诺。

#### 1. 凭证类型（货架化配置）

系统提供标准化的凭证提交入口，包括但不限于：

- **代码/技术类**：GitHub Pull Request 链接、代码仓库快照。
    
- **设计/文档类**：Figma 协作链接、Notion 文档哈希、PDF 成果附件。
    
- **多媒体类**：工作成果截图、录屏视频（由系统自动生成时间戳）。
    
- **物流/实物类**：第三方物流单号、地理位置签名。
    

#### 2. 存证与私密性

- 所有凭证在上传时生成唯一哈希（Hash），确保存证不可篡改。
    
- 公会通过加密技术保障凭证内容仅协作双方可见，但哈希值公开用于信用审计。
    

### 5.4 信用反馈逻辑 (Credit Feedback)

任务协作是 VCP 增长的唯一“燃料”。当任务进入“结算态”时，系统将触发以下信用反馈动作：

- **信用锚定**：根据任务完成的金额、难度和客户评价，系统计算本次应得的 VCP 分值。
    
- **自动铸造触发**：任务状态的变更会向第六部分的“协议层”发送一个铸造信号。
    
- **历史归档**：该任务的成功履约将永久记录在成员的“职业足迹”中，成为其进入高级频道或获取勋章的依据。
    

### 5.5 争议处理与公会仲裁 (Guild Arbitration)

当“代码”无法解决人的主观分歧时，引入公会治理逻辑。

- **仲裁发起**：任何一方在“验收态”无法达成一致时，可申请介入。
    
- **随机评审团**：系统根据第六部分定义的治理规则，随机邀请持有“提灯人”或高 VCP 值的成员组成临时仲裁组。
    
- **证据查阅**：仲裁组仅有权查阅该里程碑关联的“任务凭证”。
    
- **强制执行**：仲裁结果直接决定托管资金的分配比例，并记录恶意违约者的“信用污点”。

### 5.6 给开发团队的业务逻辑指令

在开发这一部分时，请确保：

1. **业务与协议解耦**：前端和后端只需维护“任务状态”和“凭证文件”。一旦涉及到“钱”和“信誉分”的变动，必须通过第六部分定义的标准接口向链上发送指令。
    
2. **实时通知系统**：任务状态的每一个流转（如“对方已提交凭证”）必须通过 Webhook 实时通知相关方。
    
3. **可视化终端**：任务进度管理界面应保持 3D HUD 风格，每一个里程碑的解锁应有明确的视觉反馈（如立方体的点亮或光流转）。
    

## 6.协议层与链上逻辑技术规范 (Blockchain & Protocol)

### 6.1 链上架构全景 (On-chain Landscape)

本项目采用“协议驱动应用”的模式。所有核心业务逻辑的“真值”均存储于区块链，前端 HUD 界面仅作为协议的交互终端。

- **信用底座**：已发行的 VCP (SBT) 合约。
    
- **权益中枢**：勋章 NFT 合约（特权开关）。
    
- **金融中枢**：GuildEscrow 托管合约（资金与生息管理）。
    
- **决策中枢**：治理引擎合约（权力分配与执行）。

由于 VCP 代币已在链上运行，本项目将其视为公会的“职业基因”。

    2. **全局只读**：系统内所有“信用分”展示均须实时调用该合约的 `balanceOf` 函数，严禁使用链下数据覆盖。

### 6.4 协作托管合约 (GuildEscrow) 交互逻辑

这是实现“零费率”和“信用反哺”的执行层。

#### 1. 资金与生息工作流

- **Deposit (USDT/USDC)**：用户发起任务，资金进入托管合约。
    
- **Aave Integration**：合约自动将资金通过 `Supply()` 存入 **Aave V3** 产生 `aToken` 收益。
    
- **Zero-Fee Release**：
    
    - 验证里程碑凭证后，合约将 **100% 本金** 释放给贡献者。
        
    - 托管期间产生的利息（Interest）自动划转至公会 DAO 财库。
        

#### 2. 跨合约信用触发 (SBT Minting Trigger)

在释放本金的同时，`GuildEscrow` 必须发起内部调用：

Solidity

```
// 伪代码参考
function finalizeTask(address hunter, uint256 amount) external {
    require(msg.sender == ARBITRATOR || msg.sender == TASK_OWNER);
    
    // 1. 计算基础 VCP 奖励
    uint256 baseVCP = calculateVCP(amount);
    
    // 2. 检查勋章特权 (查询 Medals NFT 合约)
    if (MedalNFT.hasPioneer(hunter)) {
        baseVCP = baseVCP * 105 / 100; // 5% 加成
    }
    
    // 3. 调用已发行的 VCP 合约进行铸造
    VCP_CONTRACT.mint(hunter, baseVCP);
}
```

### 6.5 链上治理与投票引擎 (Governance)

公会权力完全由 VCP 余额和勋章持有情况动态计算。

- **权重计算 (Snapshot)**：
    
    - 每次投票发起时，合约对用户的 VCP 余额进行快照。
        
    - 若用户持有【提灯人】，其快照余额在计算权重时自动乘 1.1。
        
- **提案门槛 (The Gate)**：
    
    - 发起提案需持有 > 500 VCP。
        
    - 若用户持有【初火】，系统在执行 `require` 检查时强制通过。
        
- **自动化执行**：投票通过后，经 48 小时时间锁（Timelock），合约自动执行参数修改（如货架 SKU 调整）。
    

---

### 6.6 数据索引与状态同步 (The Bridge)

为了保证 HUD 界面的流畅度，必须建立高性能的数据网关：

- **Subgraph 索引**：部署专门的 Subgraph，实时索引已发行 VCP 合约的 `Mint` 事件。
    
- **Supabase 同步**：后端监听 Subgraph Webhook。当链上产生一笔 VCP，前端 3D 看板的数值必须在 **3 秒内** 完成流转动效，无需用户刷新。
