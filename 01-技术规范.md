# Super Guild (The Shelter Program) 全栈技术规范文档 v1.0

## 1. 系统架构总览 (System Architecture)

本系统采用 **Web2.5 混合架构模式**，旨在平衡 Web2 的极致响应速度与 Web3 的去中心化信任。

- **交互层 (Frontend)**: Next.js 15 (App Router) + Tailwind CSS v4 + React Three Fiber。

- **后端层 (Backend)**: Supabase (PostgreSQL) + Edge Functions。

- **协议层 (On-chain)**: 以太坊 L2 (Base/Polygon) 智能合约 + 已发行的 VCP (ERC-5192)。

- **索引层 (Indexing)**: The Graph (Subgraph) + Supabase Realtime。

---

## 2. 前端交互引擎 (The Console Engine)

前端不仅是页面，而是一个“交易终端”。

### 2.1 核心技术栈

- **Next.js 15**: 利用 Server Components 减少客户端 JS 负载，使用 Server Actions 处理表单提交。

- **Tailwind CSS v4**: 利用其原生 3D 变换（3D Transforms）实现 HUD 界面。

- **Wagmi v3 + Viem**: 类型安全的智能合约交互。

- **TanStack Query v5**: 管理链下数据的缓存与异步状态。

### 2.2 3D HUD 实现规范

- **装饰件**: 首页立方体使用 `Three.js` 渲染，通过 `instancedMesh` 优化性能。

- **响应式**: 必须兼容 4K 宽屏显示，采用极简工业风（Border 1px, Zinc-800）。

---

## 3. 后端与数据工程 (The Hub)

利用 Supabase 提供的“Serverless”能力，实现高性能的中后台与任务管理。

### 3.1 数据库模式 (Schema)

- **`users` 表**: 存储 `wallet_address` (PK), `vcp_mirror_balance` (链上数据的同步镜像), `invite_code`。

- **`back_office_shelf` 表**: 存储货架 SKU。包含 `sku_id`, `category`, `metadata_json` (支持随时增加服务品类)。

- **`collaborations` 表**: 存储任务状态、里程碑、加密的凭证哈希。

- **`purchase_records` 表**: 记录中后台服务的购买权限。

### 3.2 行级安全 (RLS) 策略

- **原则**: 平台无法篡改用户数据。所有修改请求必须通过 `auth.uid() = wallet_address` 验证，确保只有私钥持有者能更新自己的任务状态。

## 4. 协议层集成规范 (Protocol & On-chain)

重点解决已发行 VCP Token 的自动化交互。

### 4.1 已发行 VCP 的调用逻辑

- **合约地址**: `${VCP_ADDRESS}` (通过环境变量注入)。

- **铸造权限**: 区块链工程师需将 VCP 合约的 `MINTER_ROLE` 授予 `GuildEscrow`（托管合约）。

- **数据读取**: 统一使用 `useContractRead` 获取实时分值。

### 4.2 GuildEscrow (任务托管合约)

- **功能**: 实现 USDT 锁定 -> Aave V3 质押生息 -> 本金释放。

- **事件抛出**: 释放本金时必须抛出 `MilestoneSettled(address hunter, uint256 vcpAmount)` 事件，供 Subgraph 监听。

---

## 5. 关键工作流细化 (Key Workflows)

### 5.1 中后台服务购买流 (Shelf Service Flow)

1. 用户点击货架卡片 -> 前端调用合约支付 USDT。

2. 支付成功事件被监听 -> Supabase 在 `purchase_records` 为该用户解锁对应的服务 SKU。

3. 用户访问服务 -> 系统检查 `purchase_records` 权限，成功后渲染 Markdown 内容。

### 5.2 任务协作：凭证验证与信誉增发流

1. **提交**: 执行者上传凭证到 Supabase Storage -> 数据库记录哈希值。

2. **验收**: 发起者在 HUD 界面点击“验收” -> 触发 `GuildEscrow` 的 `confirmMilestone` 函数。

3. **上链**: 托管合约将本金释放，并异步调用 VCP 合约的 `mint` 函数。

4. **反馈**: Subgraph 监听到 `Mint` 事件 -> 通知 Supabase 更新镜像余额 -> 用户界面 VCP 数值跳变。

## 6. 开发环境与 DevOps

- **语言**: 强类型 TypeScript (Strict Mode)。

- **合约开发**: Hardhat / Foundry。

- **部署**: Vercel (前端) + Supabase Cloud (后端) + Alchemy (RPC 节点)。

- **监控**: Sentry (前端报错) + Tenderly (合约运行监控)。

## 7. 国际化与前端规范 (i18n & Frontend Guidelines)

前端内容的展示及文案必须严格遵循双语规范：

- **强制使用语言变量**: 全站所有文本内容（包括按钮、菜单、说明等）必须通过 `lib/i18n` 中定义的字典获取，严禁在组件代码中硬编码任何中英双语文案。
- **纯净的语言展示**: 不要在界面元素中同时混合展示双语（如：“公告板 (Announcements)”），应通过 Navbar 提供的语言切换按钮（`setLocale`），由系统统一控制当前显示为纯中文或纯英文。
- **一致性**: `lib/i18n/zh.ts` 与 `en.ts` 的结构必须严格保持一致（通过 TypeScript 的 `TranslationKeys` 保证类型安全）。新增页面或组件时，必须首先在 i18n 系统中定义文案字典。
