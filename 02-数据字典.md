
# Super Guild 数据库字典 (Database Schema) v1.0

## 1. 身份与信用板块 (Identity & Credit)

这一部分是已发行 VCP Token 在应用层的镜像，用于 HUD 界面的极速渲染。

### 表：`profiles` (用户档案)

|**字段名**|**类型**|**说明**|**约束**|
|---|---|---|---|
|`wallet_address`|String (PK)|用户钱包地址（全系统唯一 ID）|Primary Key, Lowercase|
|`username`|String|用户公开昵称|Unique|
|`bio`|Text|个人简介/超级个体描述|-|
|`vcp_cache`|Numeric|链上 VCP 余额的本地镜像（由 Subgraph 同步）|Default: 0|
|`created_at`|Timestamp|注册（首次绑定钱包）时间|-|

### 表：`user_medals` (勋章持有关系)

|**字段名**|**类型**|**说明**|**约束**|
|---|---|---|---|
|`id`|UUID (PK)|关系唯一标识|-|
|`user_address`|String|关联 `profiles.wallet_address`|FK|
|`medal_type`|Enum|PIONEER (拓世者) / LANTERN (提灯人) / FLAME (初火)|-|
|`token_id`|String|该勋章在链上合约中的 TokenID|Unique|
|`minted_at`|Timestamp|链上铸造时间|-|

## 2. 万能中后台板块 (Universal Back-office)

采用“货架模式”设计，支持通过修改 `services` 表动态增加服务品类。

### 表：`services` (服务货架清单)

|**字段名**|**类型**|**说明**|**约束**|
|---|---|---|---|
|`id`|UUID (PK)|服务唯一标识|-|
|`channel`|Int|1:基础设施 / 2:专项服务 / 3:专家咨询|-|
|`category`|String|栏目名称（如：法律合规、财税统筹）|-|
|`title`|String|方案/手册名称|-|
|`price`|Numeric|售价（单位：USDT）|-|
|`unlock_type`|Enum|CATEGORY (按栏目解锁) / ITEM (按方案解锁)|-|
|`payload_config`|JSONB|配置项（存储 Markdown 内容、下载链接或表单字段定义）|-|
|`is_active`|Boolean|是否上架|Default: true|

### 表：`service_access` (服务解锁记录)

|**字段名**|**类型**|**说明**|**约束**|
|---|---|---|---|
|`id`|UUID (PK)|记录 ID|-|
|`user_address`|String|关联 `profiles.wallet_address`|FK|
|`target_id`|UUID|关联 `services.id` 或类目标识|-|
|`tx_hash`|String|支付该服务费用的链上交易哈希|Unique|

## 3. 任务协作管理板块 (P2P Collaboration)

负责管理成员之间的自发互动。

|**字段名**|**类型**|**说明**|**约束**|

|`initiator_id`|String|发起人地址|FK|

|`status`|Enum|OPEN / LOCKED / ACTIVE / PENDING / SETTLED / DISPUTED|-|
|`escrow_address`|String|该任务对应的链上托管合约地址|-|

### 表：`milestones` (里程碑明细)

|**字段名**|**类型**|**说明**|**约束**|

|**字段名**|**类型**|**说明**|**约束**|

## 4. 开发逻辑与安全约束 (Engineering Constraints)

### 4.1 链上/链下同步逻辑 (The Mirroring)

- **VCP 镜像**：开发团队必须在 `profiles` 表上设置一个数据库 Webhook。当 Subgraph 监听到已发行的 VCP 合约有 `Transfer` (Mint) 事件时，自动调用 Edge Function 更新 `vcp_cache`。
    
- **状态强一致性**：`collaborations` 表的 `status` 字段必须与 `GuildEscrow` 智能合约的状态机完全同步。
    

### 4.2 数据完整性

- **凭证不可逆**：`proofs` 表禁止 `DELETE` 和 `UPDATE` 操作。如果提交错误，执行者只能通过新增一条记录并标注为“修正版本”来覆盖，以保证信用记录的真实性。
    
- **货架动态性**：`services` 表中的 `payload_config` 使用 JSONB 格式。这允许你在不改变数据库结构的情况下，为“专家咨询”频道添加预约时间表，或为“法务服务”添加动态合同表单。
    

### 4.3 隐私保护

- 虽然所有 VCP 和任务状态在链上公开，但 `collaborations` 表中的具体沟通细节和敏感凭证应通过 Supabase 的 **RLS (行级安全)** 设置：`auth.uid() = initiator_id OR auth.uid() = provider_id`，确保只有当事人能查看具体交付物。

### 给开发团队的说明：

> “这套 Schema 是 **Super Guild** 的心脏。请注意，`profiles` 表中的 `vcp_cache` 仅仅是缓存。在涉及治理表决权或获取‘初火’勋章特权时，系统**必须**实时通过智能合约接口（Wagmi）校验链上数据，不得仅依赖数据库缓存。”
