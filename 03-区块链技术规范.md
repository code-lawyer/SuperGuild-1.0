# Super Guild 2026 协议层技术规范书 (Arbitrum 主权版)

## 1. 基础设施选型：为何选择 Arbitrum One

在 2026 年的 L2 竞争中，我们选择 **Arbitrum One** 作为物理底座，而非 Base 或 Polygon，理由如下：

- **Stage 1 状态认证**：Arbitrum 是少数实现 **BoLD (Bounded Liquidity Delay)** 的 Layer 2，支持无许可的状态挑战。这意味着即便 Offchain Labs 消失，用户依然可以通过以太坊主网强制提取资产。

- **Stylus 高性能引擎**：支持使用 Rust 编写智能合约。对于未来复杂的"VCP 权重算法"和"3D 元数据计算"，Stylus 的 Gas 效率比传统 Solidity 高出 10 倍以上。

- **抗审查性**：作为不依附于特定上市公司的去中心化网络，其排序器去中心化路线图更符合我们的"避难所"愿景。

> **当前状态**: 测试阶段部署在 **Arbitrum Sepolia** 测试网（Chain ID: 421614）。VCP 合约和 MedalNFT 合约均运行于此链。主网迁移将在业务逻辑验证完成后执行。

## 2. 资产标准规范 (Asset Standards)

### 2.1 VCP (信誉积分) —— 改良 ERC-20 (Locked, Non-transferable)

VCP 采用**改良 ERC-20 + Votes + UUPS 可升级**架构。保留 ERC-20 接口以兼容所有钱包和区块浏览器的余额显示，同时通过 `locked()` 函数强制禁止转账。

- **标准兼容**: 合约实现了 ERC-20 接口（`balanceOf`, `totalSupply` 等），但 `transfer()` 和 `approve()` 均为 `pure` 函数直接 revert，实现不可转让语义。

- **核心逻辑**：

  - `locked()`: `pure` 函数，始终返回 `true`，声明资产不可流通。

  - `mint(to, amount, reason)`: 仅限持有 `MINTER_ROLE` 的地址调用，`reason` 参数记录铸币原因（如 `"Settlement: xxxx"` 或 `"Pioneer: CODE"`）。

  - `getVotes(account)` / `delegate()`: 基于 OpenZeppelin Votes 扩展，支持投票权委托，为 DAO 治理留出接口。

  - `burn(from, amount)`: 管理员销毁功能。

  - `blacklisted(address)`: 黑名单机制，可阻止恶意地址。

- **可升级性**: 基于 UUPS 代理模式 (`proxiableUUID`)，合约逻辑可在不变更地址的情况下升级。初始化通过 `initialize(coldWallet, hotWallet)` 完成，分离冷热钱包职责。

- **合约地址**: `0xcDD2b15fEFC2071339234Ee2D72104F8E702f63C` (Arbitrum Sepolia)

> **关于 ERC-8129**: 原规划采用 ERC-8129 标准。评估后认为当前阶段工具链支持不成熟，改良 ERC-20 + locked 方案等效实现不可转让语义，且兼容性更好。待 ERC-8129 生态成熟后可通过 UUPS 升级迁移。

### 2.2 NFT 资产体系

项目中存在**两套独立的 NFT 系统**，各自承担不同职责：

#### 2.2.1 特权 NFT (Privilege NFTs) — ERC-1155 (Manifold)

用于**功能门控** (Feature Gating)：控制用户对特定功能模块的访问权限。

- **合约**: Manifold ERC-1155，部署在 **Sepolia ETH**（Chain ID: 11155111）
- **地址**: `0x46486Aa0aCC327Ac55b6402AdF4A31598987C400`
- **查询方式**: `balanceOfBatch` 批量检查持有状态

| Token ID | 英文名 | 中文名 | 用途 |
|:---------|:-------|:-------|:-----|
| 1 | Pioneer Memorial | 拓世者纪念章 | 创始成员纪念标识 |
| 2 | Lantern Keeper's Withered Lamp | 提灯人枯盏 | 社区守护者标识 |
| 3 | The First Flame | 初火 | 核心贡献者标识 |
| 4 | Hand of Justice | 公义之手 | 仲裁庭准入凭证 |
| 5 | Beacon of the Forerunner | 先驱者灯塔 | Pioneer 门控凭证 |

**门控映射**:

- 仲裁庭 (Hand of Justice 页面): 检查 Token #4
- Pioneer 专属功能: 检查 Token #5

#### 2.2.2 勋章 NFT (Medal NFTs) — ERC-721 + ERC-7496 (Dynamic Traits)

用于**3D 可进化勋章**：勋章的材质、光效等与链上 traits 关联。

- **合约**: 自定义 ERC-721 + ERC-7496，部署在 **Arbitrum Sepolia**（Chain ID: 421614）
- **地址**: `0xef96bE9fFf59B5653085C11583beaC0D16450F1a`
- **动态属性**: 合约包含 `TRAIT_GLOW`、`TRAIT_RANK` 等 trait 键值，通过 `getTraitValue(tokenId, traitKey)` 查询
- **3D 模型引用**：合约包含 `baseArweaveURI` 函数，设计为指向 Arweave 永久存储的 GLB 文件

> **当前状态**: 前端尚未集成 MedalNFT 合约的读取逻辑。3D 渲染的 trait 参数目前为硬编码，GLB 模型从本地 `/public/models/` 加载。计划在 Phase 2 完成链上 trait → 渲染参数的闭环。

## 3. 账户体系与 UX 规范 (AA & Passkey)

### 3.1 目标架构 (Phase 2)

为了让"自由"不以"难用"为代价，远期目标是实现 **全账户抽象 (Native Account Abstraction)**。

- **标准**：**ERC-4337 v2.0**。

- **登录方式**：**Passkey (WebAuthn)**。用户使用手机指纹或 FaceID 登录，系统自动在 Arbitrum 上为其创建智能合约钱包。

- **Paymaster (免 Gas 体验)**：公会设立一个国库账户。当用户执行"领取勋章"或"确认任务"等非资金交易时，Gas 由公会代付。

- **签名策略**：使用 **Session Keys**。用户在 3D 勋章墙拖拽或浏览时，无需频繁弹出钱包签名。

### 3.2 当前实现

- **钱包连接**: RainbowKit + WalletConnect，支持 MetaMask、Coinbase Wallet 等主流 EOA 钱包
- **身份同步**: 连接钱包后自动 upsert 到 Supabase `profiles` 表 (`useSyncProfile`)
- **链配置**: Wagmi config 中默认 Arbitrum Sepolia，同时配置 Sepolia ETH 用于跨链 NFT 查询

> **决策备注**: AA 体系是完整的基础设施工程，需要部署 EntryPoint、Paymaster 合约，集成 Safe/Privy SDK。在测试阶段先用 EOA 钱包验证业务逻辑是正确的工程策略。

## 4. 3D 渲染与链上数据集成 (The 3D Pipeline)

### 4.1 渲染管线

- **模型格式**：GLB (Binary glTF)。

- **交互引擎**：**React Three Fiber (R3F)**。

- **已实现组件**:
  - `BadgeModel.tsx`: 3D 勋章渲染，支持动态辉光（根据类型切换颜色）、缩略图/全尺寸模式
  - `BadgeShowcaseModal.tsx`: 全屏 3D 展示弹窗，含 OrbitControls 交互和 Environment 环境光

- **查询流程**：
    1. 用户连接钱包后，`usePrivilegeNFTs` 自动调用 `balanceOfBatch` 批量查询 5 个 Token 的持有状态
    2. 各门控 hook (`useLanternGate`, `usePioneerGate`) 通过 `useNFTGate` 对特定 Token 做单独查询
    3. 跨链查询通过 wagmi `chainId` 参数强制读取 Sepolia，用户无需切换网络

- **未实现 (Phase 2)**：
  - `ownerOf` 确权后再渲染高清模型
  - Ghost Mode（未拥有的 NFT 显示半透明锁定模型）
  - 链上 `getTraitValue` → 3D 材质参数驱动
  - KTX2 纹理压缩
  - GLB 从 Arweave 加载（当前从 `/public/models/` 本地加载）

### 4.2 数据索引

- **目标工具**：**Ponder** (事件索引框架)。

- **要求**：禁止在高频场景下直接轮询 RPC 节点。通过索引层获取 VCP 实时变动，延迟控制在 500ms 以内。

> **当前状态**: 所有链上读取通过 wagmi `useReadContract` / `useReadContracts` 直接查询 RPC。当前用户量下性能可接受。Phase 2 引入 Ponder 索引层。

## 5. AI Oracle 评估引擎

> **注意**: 此模块在原规范中未描述，但已在代码中完整实现。补充记录如下。

### 5.1 功能概述

AI Oracle 是任务结算的自动化评估系统，在任务所有里程碑确认完成后触发，负责：

1. 抓取任务提交的工作量证明 (Proof of Work, PoW)
2. 调用 LLM 进行公平评估
3. 根据评估结果在链上 mint 相应数量的 VCP

### 5.2 评估公式

```
VCP = floor(baseScore × complexityScore × qualityScore × efficiencyScore)
```

| 维度 | 范围 | 说明 |
|:-----|:-----|:-----|
| baseScore | `min(100, max(5, ln(budget) × 5))` | 由任务预算决定 |
| complexityScore | 0.5 - 2.0 | 0.5=机械复制, 2.0=前沿研究 |
| qualityScore | 0.8 - 1.2 | 0.8=勉强达标, 1.2=超预期 |
| efficiencyScore | 0.8 - 1.2 | 0.8=严重拖延, 1.2=极高效 |

### 5.3 PoW 抓取策略

| URL 类型 | 抓取方式 |
|:---------|:---------|
| GitHub PR | GitHub REST API (diff + metadata) |
| GitHub Commit | GitHub REST API (files changed) |
| GitHub Repo | README + 最近 10 commits |
| 通用 URL | Jina Reader API → Markdown |
| 兜底 | 原始 HTML 文本截取 |

### 5.4 链上铸币

- 由服务端热钱包（`HOT_WALLET_PRIVATE_KEY`）签名，调用 VCP 合约 `mint(to, amount, reason)`
- 幂等性保障: 通过 `vcp_settlements` 表检查是否已评估
- 支持 Claude / GPT / Gemini 作为评估 LLM

## 6. 创世迁移计划 (Genesis Migration)

> **状态**: 待定。取决于正式上线到 Arbitrum One 的时间点。

鉴于用户量较少，我们不采用复杂的跨链桥，而是执行 **"快照-重铸"**：

1. **Polygon 快照**：在 `Block_Height: XXXXX` 获取所有 VCP 地址清单。

2. **权限重设**：新版 Arbitrum VCP 合约保留一个 `Genesis_Migration_Role`。

3. **空投重铸**：后端脚本批量在 Arbitrum 上 mint 对应数额。

4. **前端引导**：旧用户连接钱包后，系统检测到 Polygon 资产，自动切换网络并完成账户初始化。
