我们的核心目标是**“主权自由”**。这意味着技术选型必须优先考虑抗审查性、去中心化程度以及用户对资产的绝对所有权。

# Super Guild 2026 协议层技术规范书 (Arbitrum 主权版)

## 1. 基础设施选型：为何选择 Arbitrum One

在 2026 年的 L2 竞争中，我们选择 **Arbitrum One** 作为物理底座，而非 Base 或 Polygon，理由如下：

- **Stage 1 状态认证**：Arbitrum 是少数实现 **BoLD (Bounded Liquidity Delay)** 的 Layer 2，支持无许可的状态挑战。这意味着即便 Offchain Labs 消失，用户依然可以通过以太坊主网强制提取资产。
    
- **Stylus 高性能引擎**：支持使用 Rust 编写智能合约。对于未来复杂的“VCP 权重算法”和“3D 元数据计算”，Stylus 的 Gas 效率比传统 Solidity 高出 10 倍以上。
    
- **抗审查性**：作为不依附于特定上市公司的去中心化网络，其排序器去中心化路线图更符合我们的“避难所”愿景。

## 2. 资产标准规范 (Asset Standards)

### 2.1 VCP (信誉积分) —— **ERC-8129 标准**

我们放弃 ERC-20 甚至 ERC-5192，采用 2026 年针对“非转让资产”的最优标准 **ERC-8129**。

- **技术特性**：完全剔除 `transfer` 逻辑。相比 ERC-721 的魔改版，其字节码缩减 80%，极大降低了在大规模任务结算时的 Gas 消耗。
    
- **核心逻辑**：
    
    - `locked()`: 默认返回 `true`。
        
    - `mint()`: 仅限 **协作托管合约** 拥有 `MINTER_ROLE`。
        
    - `reputation_weight()`: 返回基于 VCP 与时间的动态加权值。
        

### 2.2 3D 勋章 —— **ERC-7496 (Dynamic Traits) 标准**

勋章不仅是图片，而是包含 3D 状态的可进化资产。

- **动态属性**：利用 ERC-7496，勋章的 3D 材质（如光效、磨损度）直接与链上的 `traits` 关联。
    
- **3D 模型引用**：`tokenURI` 指向 **Arweave** (永久存储) 上的 GLB 文件哈希。
    
- **确权机制**：前端渲染器在读取 GLB 之前，必须调用合约验证 `ownerOf`。

## 3. 账户体系与 UX 规范 (AA & Passkey)

为了让“自由”不以“难用”为代价，我们必须实现 **全账户抽象 (Native Account Abstraction)**。

- **标准**：**ERC-4337 v2.0**。
    
- **登录方式**：**Passkey (WebAuthn)**。用户使用手机指纹或 FaceID 登录，系统自动在 Arbitrum 上为其创建智能合约钱包。
    
- **Paymaster (免 Gas 体验)**：公会设立一个国库账户。当用户执行“领取勋章”或“确认任务”等非资金交易时，Gas 由公会代付。
    
- **签名策略**：使用 **Session Keys**。用户在 3D 勋章墙拖拽或浏览时，无需频繁弹出钱包签名。

## 4. 3D 渲染与链上数据集成 (The 3D Pipeline)

### 4.1 渲染管线

- **模型格式**：GLB (Binary glTF) + KTX2 压缩纹理（确保 3D 弹窗在移动端秒开）。
    
- **交互引擎**：**React Three Fiber (R3F)**。
    
- **逻辑流**：
    
    1. 用户点击“勋章墙” -> 触发 `wallet_scan`。
        
    2. 调用合约 `balanceOfBatch` 批量查询勋章所有权。
        
    3. **所有权检查**：
        
        TypeScript
        
        ```
        const isOwner = await contract.ownerOf(tokenId) === userAddress;
        if (isOwner) {
            renderHighRes3D(modelUrl); // 加载 4K 贴图 GLB
        } else {
            renderGhostMode(); // 仅展示半透明锁定模型
        }
        ```
        

### 4.2 数据索引

- **工具**：**Goldsky / Ponder**。
    
- **要求**：禁止直接轮询 RPC 节点。必须通过索引层获取 VCP 实时变动，延迟必须控制在 500ms 以内，以保证 3D 交互的即时反馈。

## 5. 创世迁移计划 (Genesis Migration)

鉴于用户量较少，我们不采用复杂的跨链桥，而是执行 **“快照-重铸”**：

1. **Polygon 快照**：在 `Block_Height: XXXXX` 获取所有 VCP 地址清单。
    
2. **权限重设**：新版 Arbitrum VCP 合约保留一个 `Genesis_Migration_Role`。
    
3. **空投重铸**：后端脚本批量在 Arbitrum 上 mint 对应数额。
    
4. **前端引导**：旧用户连接钱包后，系统检测到 Polygon 资产，自动切换网络并完成账户初始化。
