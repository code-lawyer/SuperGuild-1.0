## 一、摘要

SuperGuild（超级公会），副标题"庇护所计划"（The Shelter Program），是一个面向 AI 时代超级个体的**去中心化协作协议与基础设施平台**。

在传统劳务协作平台上，中间商抽取高额佣金、规则不透明、争议处理缺乏公信力，协作者的声誉更无法跨平台携带。SuperGuild 以智能合约取代中心化管理员，以链上声誉取代平台私有评级，以 DAO 自治取代单边规则制定——构建一个**真正属于协作者自己**的去许可基础设施。

SuperGuild 的核心创新体现在四个支柱：

1. **零费率协作协议** — 基于 USDC 稳定币的直接 P2P 结算，平台不抽取任何中间费用
2. **链上声誉铸造** — VCP（Vibe Contribution Point）作为不可转让的贡献凭证，永久记录在 Arbitrum 上
3. **乐观惩罚仲裁** — 7 天争议窗口 + Hand of Justice 链上投票，博弈机制替代人工审核
4. **万能中后台** — 货架化法律/财税/IP/跨境服务，稳定币一站解锁，消除超级个体的运营壁垒

SuperGuild 现已在 Arbitrum Sepolia 测试网完成全系统部署，进入公测阶段。

## 二、背景与问题

### 2.1 传统协作平台的结构性困境

当前主流自由职业与协作平台（Upwork、Fiverr、猪八戒等）均存在以下系统性问题：

**① 高额中间费用**
平台对每笔交易收取 5%–20% 的佣金。以年交易额 10 万美元的独立开发者为例，每年需向平台缴纳 1–2 万美元，但这笔费用对应的服务——信任担保、争议仲裁——质量却参差不齐。

**② 声誉数据封闭、不可携带**
协作者在平台上积累的评分、完成率、历史记录，全部由平台单向持有。一旦平台政策变动、账号被封或平台关闭，数年积累的信誉资产瞬间归零。声誉的核心价值——信任——却完全掌握在第三方手中。

**③ 争议处理缺乏可信公正性**
中心化平台的争议仲裁本质上是平台单方裁定。仲裁规则不透明、执行不一致，协作双方都可能遭受不公正的结果，且无任何上诉机制。

**④ 运营壁垒高企**
超级个体——那些以 AI 工具武装的独立开发者、创作者和战略顾问——在处理核心创造性工作之外，还需独立应对：公司注册、税务合规、IP 保护、跨境收款、合规合同。这些非创造性工作消耗了大量精力，却无法产生直接价值。

### 2.2 Web3 协作赛道的现有尝试与局限

部分 Web3 协作平台（Dework、Layer3 等）尝试以链上任务系统解决部分问题，但仍存在明显局限：

- 主要面向加密原生的 DAO 贡献奖励场景，缺乏通用的商业协作支撑
- 代币化激励易引发刷单与 Sybil 攻击，缺乏有效的反作弊机制
- 支付以协议原生代币为主，波动性大，不适合作为商业结算媒介
- 缺乏完整的争议解决机制，社会治理的博弈设计缺失

SuperGuild 正是在认识到上述不足之后，选择以 **USDC 稳定币为结算媒介**、以**可验证的链上行为**为声誉基础、以**博弈论设计**为仲裁内核，构建一套真正可用于商业协作的去中心化协议。

## 三、核心理念：超级个体时代已到来

### 3.1 AI 武装的超级个体

大型语言模型的普及正在重塑劳动力市场的基本单元。过去需要一个 10 人团队完成的工作——产品设计、开发、运营、法律、财务——现在一个掌握 AI 工具链的个体即可独立承接。这类个体被称为**超级个体**（Super Individual）：

- 技术敏感，习惯高效工具，适应异步工作
- 高度自主，不依赖机构身份，以作品和声誉建立信任
- 链上原住民，以钱包地址作为自己的数字主权声明
- 拒绝中间商剥削，追求对等、直接、可验证的协作关系

### 3.2 超级个体的核心需求

| 需求 | 现有方案的缺陷 | SuperGuild 的回应 |
|------|--------------|-----------------|
| 安全收取报酬 | 平台抽成高、跨境汇款周期长 | USDC 零费率直接结算 |
| 建立跨平台可信声誉 | 声誉被平台封闭持有 | 链上 VCP 不可伪造、不可删除 |
| 公平解决协作争议 | 平台单边仲裁 | Hand of Justice 去中心化投票仲裁 |
| 应对非创造性运营壁垒 | 各类服务分散且昂贵 | 万能中后台货架化服务 |
| 参与决策、共建规则 | 用户无治理权 | DAO 提案与 VCP 权重投票 |

### 3.3 "赛博避难所"的文化隐喻

SuperGuild 的定位不是冷酷的金融工具，也不是热闹的社交平台。它是一个**严肃但有温度的协作基础设施**——可信赖、可验证、对等。

"避难所"意味着：当一个超级个体从传统组织结构或中心化平台出走，进入去中心化的协作世界，这里应该成为他的主场——有归属感的成员身份、可携带的链上声誉、平等参与的治理权力。

## 五、协作协议：零费率 P2P 结算

### 5.1 核心设计原则

SuperGuild 的协作协议有三个不可妥协的原则：

1. **零平台佣金** — 协作报酬 100% 流向劳动者，协议层不抽取任何中间费用
2. **USDC 稳定结算** — 以 Circle USDC 作为结算媒介，规避代币价格波动对商业协作的干扰
3. **链上可验证** — 关键协作行为（支付、里程碑确认）在 Arbitrum One 链上留有不可篡改的记录

### 5.2 协作状态机

每个协作任务经历以下生命周期状态：

```
OPEN
  → PENDING_APPROVAL（有人申请）
  → ACTIVE（发布者批准）
  → 里程碑循环（提交凭证 → 确认 → 支付）
  → SETTLED（全部里程碑完成）

任意阶段 → CANCELLED（发布者主动撤销）
ACTIVE   → DISPUTED（guild_managed 模式，发布者发起争议）
```

### 5.3 里程碑制度

协作任务以**里程碑**为最小执行单元。每个里程碑对应一个交付节点和资金占比：

- S/A 级任务强制 ≥ 3 个里程碑，防止大额任务单次支付风险
- 每个里程碑独立确认，资金按比例释放
- 里程碑状态：`INCOMPLETE → SUBMITTED → CONFIRMED`

里程碑子状态机确保：承接人只能提交，发布者只能确认或退回，双方均无法单边跳过环节。

### 5.4 协作等级体系

| 等级 | 名称 | VCP 基准值 | 最低里程碑数 |
|------|------|-----------|------------|
| S | 史诗 | 500 | 3 |
| A | 传奇 | 300 | 3 |
| B | 精英 | 150 | 1 |
| C | 标准 | 80 | 1 |
| D | 入门 | 40 | 1 |
| E | 体验 | 10 | 1 |

等级决定 VCP 铸造上限，高等级任务要求更严格的里程碑拆分，防止低质量刷单。

### 5.5 多人申请队列

发布者公开招募时，多名承接人可同时申请。发布者查看所有申请后选择一位批准，其余申请自动关闭。系统防止同一地址对同一协作重复申请。

## 六、链上声誉系统：VCP 积分

### 6.1 VCP 的本质

**Vibe Contribution Point（VCP）**是 SuperGuild 的链上声誉凭证，部署于 Arbitrum 上的 ERC-20 合约（VCPTokenV2，UUPS 可升级代理）。

VCP 的设计哲学与普通 ERC-20 代币有根本差异：

- **不可转让（Soulbound）** — VCP 只能被 Minter 角色铸造，持有者无法自由转让，声誉不可交易
- **不可伪造** — 每次铸造均对应链上可验证的协作结算事件（DirectPay.Paid 或 GuildEscrow.MilestoneSettled）
- **永久可查** — 历史铸造记录永久保存在 Arbitrum 上，任何人可验证某地址的贡献历史

### 6.2 VCP 铸造机制

VCP 通过 **Resolver Bot**（热钱包自动化服务）在协作完成后自动铸造：

```
DirectPay.Paid 事件 (self_managed)
    → Resolver Bot 监听
    → 反作弊检查通过
    → 调用 VCPTokenV2.mint(worker, amount × 0.5x, reason)
    → 链上铸造记录

GuildEscrow.MilestoneSettled 事件 (guild_managed)
    → Resolver Bot 监听
    → 反作弊检查通过
    → 调用 VCPTokenV2.mint(worker, amount × 1.0x, reason)
    → 链上铸造记录
```

**双轨乘数**体现对信任等级的区分：guild_managed 模式通过智能合约托管资金、链上存证，协作质量更可信，因此享有 1.0x 完整倍率；self_managed 模式为无托管直通，享有 0.5x 倍率。

### 6.3 反作弊保护机制

VCP 的价值依赖于其真实性。SuperGuild 实现了三层自动反作弊：

**① 7 天冷却期**
同一发布者与承接人组合，7 天内只能触发一次有效 VCP 铸造。防止双方循环协作刷积分。

**② 月度上限**
单地址每月可获得的 VCP 上限为 1,000 点。超出部分自动跳过铸造，不累积到下月。

**③ 快速确认冻结**
监测最近 10 笔结算中，若 ≥ 5 笔的"凭证提交"到"确认支付"时间间隔 < 120 秒，视为异常快速确认（正常协作不可能几分钟内完成），触发 30 天 VCP 冻结。

所有反作弊决策及跳过原因均写入 `vcp_settlements` 表，链上记录（交易哈希）和链下分析数据双重留存，保持完整的审计追踪。

### 6.4 VCP 的治理权重

在 DAO 投票中，VCP 余额决定投票权重。拥有更多链上贡献历史的成员，对协议规则拥有更大的发言权——这确保了治理话语权与真实贡献之间的正相关。

## 七、乐观惩罚机制与仲裁庭

### 7.1 乐观释放设计

GuildEscrow 合约采用**乐观释放（Optimistic Release）**模式——默认假设协作双方诚实行事：

```
承接人提交凭证（submitProof）
  → 开始 7 天倒计时
  ├── 发布者主动确认 → 立即释放 USDC
  ├── 7 天无操作 → autoRelease() 自动释放（热钱包代调，平台承担 Gas）
  └── 发布者发起争议 → 进入仲裁流程
```

这一设计的核心洞察是：大多数协作是诚实的。设置"默认通过"的乐观逻辑，减少正常协作的摩擦；仅在异常时启动仲裁，而非所有协作都需要人工审核。

### 7.2 Hand of Justice 仲裁庭

争议触发后，由持有 **Token #4（Hand of Justice）**NFT 的仲裁员进行链上投票。NFT 门控确保仲裁资格有明确的链上准入门槛，而非由平台单边指派。

仲裁结果执行：
- 仲裁员投票支持承接人（Worker Wins）→ USDC 全额释放给承接人
- 仲裁员投票支持发布者（Publisher Wins）→ USDC 退还发布者，10% 作为惩罚没入国库

**10% 惩罚机制**的博弈论意义：使恶意发起争议（拒绝支付已完成工作）在预期上不合理——败诉方需额外损失 10%，对诚实的发布者无额外成本，对恶意方形成威慑。

### 7.3 去中心化仲裁的价值

| 维度 | 传统平台 | SuperGuild 仲裁庭 |
|------|---------|-----------------|
| 仲裁者 | 平台客服（匿名） | 链上可验证的 NFT 持有者 |
| 规则 | 平台私有，随时可变 | 智能合约代码，公开不可篡改 |
| 结果执行 | 平台单边操作 | 合约自动执行，无需信任任何人 |
| 可信度 | 依赖平台声誉 | 依赖代码和密码学 |

## 八、万能中后台：超级个体的数字基座

### 8.1 定位

超级个体最大的非核心时间消耗来自**运营后勤**：理解适用法律、处理财税申报、保护知识产权、对接跨境业务。这些工作本质上是标准化的，但对个人而言成本极高。

万能中后台将这些服务**货架化**——标准化定价、USDC 一键解锁、即时交付或预约对接，让超级个体专注于创造本身。

### 8.2 三频道架构

**频道一：基础设施**（`/services/infrastructure`）
面向超级个体数字经营基座的工具和文档服务：合规模板合同、隐私协议框架、数字资产保护指南等。稳定币链上激活，解锁后文件永久可下载。

**频道二：核心服务**（`/services/core`）
专项商业解决方案，覆盖法律合规、财税统筹、IP 防御、跨境贸易、自媒体运维等分类。以 Calendly 预约或联系方式对接专业顾问。

**频道三：专家咨询**（`/services/consulting`）
一对一专家资源对接。专家展示技能标签、服务费率和预约方式，买方直接预约，中间无佣金截留。

### 8.3 支付模型

基础设施频道采用链上支付：ERC-20 USDC `transfer` 至协议国库，智能合约事件作为解锁凭证。核心服务和专家咨询则通过配置化的外链（Calendly、联系二维码）完成对接，协议层仅作路由。

## 九、DAO 治理：去中心化自治

### 9.1 治理设计哲学

SuperGuild 的 DAO 治理基于以下前提：**影响最深的人拥有最大的发言权**。

VCP 作为链上贡献的累积凭证，天然成为治理权重的基础。平台规则的修改、服务费率的调整、仲裁阈值的变更——都需要经过 VCP 加权投票，而非中心化团队单边决定。

### 9.2 提案生命周期

```
创建提案（DRAFT）
  → 联署阶段（GATHERING_COSIGNS）—— 需达到联署门槛
  → 投票激活（ACTIVE_VOTING）—— VCP 加权投票
  → 投票结束 → 执行（EXECUTED）或否决（REJECTED）
  → 归档（ARCHIVED）
  或 提案人撤回（WITHDRAWN）
```

提案类型包括：协议参数修改、新功能请求、国库资金使用、成员特权授予/撤销等。

### 9.3 柴薪王座（治理前台）

`/council/proposals` 是 DAO 治理的可视化界面。每笔提案的投票记录、投票权重、状态历史均公开可查。治理过程完全透明，不存在黑箱决策。

## 十、身份与特权系统：Privilege NFT

### 10.1 设计逻辑

SuperGuild 使用 **ERC-1155 Privilege NFT**（部署于 Ethereum Mainnet，Manifold 铸造）作为特权门控系统。NFT 持有即资格证明，合约代码即规则——无法伪造，无法被平台单边撤销。

跨链查询通过 Wagmi + Alchemy NFT API 双源验证机制实现：主 RPC 失败时自动切换 HTTP 后备查询，且任何错误均遵循 **Fail-closed 原则**（宁可误报无权限，绝不在错误时放行）。

### 10.2 五枚勋章的体系

| Token ID | 名称 | 意义 | 功能门控 |
|----------|------|------|---------|
| #1 | Pioneer Memorial | 最早期社区创始成员 | 创始纪念，无特殊功能门控 |
| #2 | Lantern Keeper's Withered Lamp | 社区长期守护者 | 社区标识 |
| #3 | The First Flame | 协议管理员 | Admin 面板准入 |
| #4 | Hand of Justice | 仲裁庭成员 | 仲裁投票资格 |
| #5 | Beacon of the Forerunner | Pioneer 先锋成员 | 星火广场发帖权 |

### 10.3 NFT 门控的去中心化意义

传统平台的管理员资格依赖账号体系，可被平台单边撤销。SuperGuild 的管理员资格绑定在链上 NFT 上——持有 Token #3 即具备管理权，转移 NFT 即转移权力，任何一方（包括协议开发者）均无法通过修改数据库来撤销这一权利。

这是从"许可制"向"无许可制"迈进的关键一步。

### 11.2 智能合约体系

**VCPTokenV2**（Arbitrum，UUPS 可升级代理）
不可转让 ERC-20，持有声誉凭证。仅 MINTER_ROLE 可铸造，保证供给可控。UUPS 模式允许合约逻辑升级，同时保持代理地址不变，确保现有持有者无需迁移。

**GuildEscrow**（Arbitrum，单一共享合约）
一个合约服务所有协作，以 `collabId`（协作 UUID 的 keccak256 哈希）区分各协作资金池。核心函数：
- `deposit(collabId)` — 发布者锁仓 USDC
- `submitProof(collabId, milestoneIdx, contentHash)` — 承接人提交凭证，启动 7 天倒计时
- `confirmMilestone(collabId, milestoneIdx)` — 发布者确认，立即释放
- `autoRelease(collabId, milestoneIdx)` — 热钱包在 7 天后代调，乐观释放
- `disputeMilestone(collabId, milestoneIdx)` — 发布者发起争议
- `resolveDispute(collabId, milestoneIdx, workerWon, penalizeLoser)` — 仲裁执行

**DirectPay**（Arbitrum，无状态转发器）
MVP 阶段的轻量级支付路由。发布者调用 `pay(collabId, worker, amount)`，合约原子性地将 USDC 从发布者转至承接人，同时触发 `Paid(collabId, publisher, worker, amount)` 事件供 Resolver Bot 索引。攻击面接近于零。

**MedalNFT**（Arbitrum，ERC-721 + ERC-7496 Dynamic Traits）
成就勋章系统，支持动态特征更新。链上特征参数未来可驱动 3D 渲染差异化。

### 11.3 Supabase 数据模型

核心业务表结构：

| 表 | 职责 | 写入保护 |
|----|------|---------|
| `collaborations` | 协作主表，状态机核心 | RLS：只有 initiator 可创建/更新特定字段 |
| `milestones` | 里程碑明细 | RLS：只有协作双方可读写 |
| `proofs` | 凭证链（**禁止 DELETE/UPDATE**） | RLS：只能 INSERT，provider 有写权 |
| `collaboration_applications` | 多人申请队列 | RLS：申请人只能操作自己的记录 |
| `vcp_settlements` | VCP 铸造幂等锁 | RLS：只有 service_role 可写 |
| `notifications` | 站内通知 | RLS：只能读写自己的通知 |
| `profiles` | 用户档案 | RLS：只能修改自己的档案 |

`proofs` 表的只追加（Append-Only）设计是有意为之：凭证历史是协作信用系统的核心，任何修改或删除都会破坏可审计性。

### 11.4 身份验证架构

```
用户操作（写请求）
  → 前端调用 /api/auth/nonce 获取一次性随机数
  → 用户钱包 EIP-191 签名（防重放：nonce + timestamp）
  → POST /api/auth/wallet（签名验证 + nonce 消耗 → 写入 auth_nonces）
  → 签发 JWT（含 wallet_address claim）
  → Supabase 代理客户端携带 JWT
  → RLS 策略通过 auth.wallet_address() 函数提取 claim
  → 数据库层面强制身份校验
```

## 十二、安全模型

### 12.1 多层防御架构

SuperGuild 的安全模型遵循**深度防御**原则，绝不依赖单一防线：

**第一层：智能合约层**
- `GuildEscrow` 中 USDC 完全由合约控制，任何操作须满足合约条件
- Resolver Bot 热钱包仅持有有限调用权（autoRelease / resolveDispute），不持有用户资金
- 合约升级需经过多签（UUPS Proxy Owner）

**第二层：API 路由层**
- 所有写操作 API 均要求 EIP-191 钱包签名验证
- Nonce 一次性消耗，防止重放攻击
- Rate Limiting 全覆盖，防止暴力攻击
- 特权 API（Admin / 仲裁 / Pioneer）额外验证 NFT 持有状态

**第三层：数据库层（Supabase RLS）**
- 全部业务表启用行级安全策略（Row Level Security）
- 身份通过 JWT claim 而非 session 变量验证，防止 JWT 伪造绕过
- `proofs` 表无 DELETE/UPDATE 权限，保证凭证不可篡改
- 敏感管理表（vcp_settlements 等）仅 service_role 可写

**第四层：前端层**
- NFT 门控双源验证（RPC + HTTP API），Fail-closed 原则
- 开发 Mock 环境变量（`NEXT_PUBLIC_DEV_MOCK_NFTS`）在生产环境强制关闭
- Content Security Policy 限制外部资源加载

### 12.2 反作弊系统的安全价值

VCP 的价值完全依赖于供给诚实性。SuperGuild 的三层反作弊（冷却期、月上限、快速确认冻结）从算法层面确保：即使攻击者控制多个协作账户，刷积分的边际收益也受到严格约束，攻击成本远高于潜在收益。

### 12.3 主网前必须完成的审计

在主网部署前，GuildEscrow 和 VCPTokenV2 合约将接受专业安全审计（目标机构：OpenZeppelin / Certik 或同级别机构）。这是对持有用户资金的合约的最低安全门槛，不可绕过。

## 十三、双轨支付架构：分阶段去信任

### 13.1 架构决策的理性

SuperGuild 采用**双轨并行**的支付架构，旨在平衡安全性与上市速度：

| | **DirectPay（自行管理）** | **GuildEscrow（公会托管）** |
|---|---|---|
| 资金托管 | 无，USDC 原子直通 | 合约全程持有 |
| 链上证据 | 仅支付金额 | 里程碑哈希 + 时间戳 |
| 7 天争议窗 | 无 | 合约强制执行 |
| 仲裁 | 不可用 | 完整（罚没 + 投票） |
| VCP 倍率 | 0.5x | 1.0x |
| 审计要求 | 极低（无托管） | 专业审计必须 |

MVP 阶段仅开放 DirectPay 模式，原因有三：
1. GuildEscrow 持有用户资金，主网上线前必须完成专业安全审计
2. DirectPay 是无状态转发，攻击面接近于零，可安全提前部署
3. 先用 P2P 模式建立社区信任，再开放托管模式

### 13.2 渐进式去信任路径

这一设计呈现了 SuperGuild 对去中心化的理解：**去信任化是一个渐进过程，不是非黑即白的选择**。

- Phase 1（当前）：协作证据存于 Supabase，仅支付记录上链
- Phase 2（审计后）：证据哈希 + 资金托管全上链，完整仲裁体系激活
- Phase 3（远期）：Arweave 永久存储，去中心化索引，链上证据完全自主

每个阶段的安全边界明确，已实现代码完整保留，随时可激活。

## 十五、总结

SuperGuild 不是另一个佣金更低的 Upwork，也不是更复杂的 Web3 积分平台。它是一次对**协作基础设施本身**的重新设计——让智能合约替代中间商、让链上声誉替代封闭评级、让 DAO 自治替代单边规则、让密码学保证替代平台信任。

**对超级个体而言**，SuperGuild 提供的是：你的每一次协作贡献都永久记录在链上，你的声誉属于你自己，任何平台都无法删除或封锁它。你的报酬通过合约直接到账，不经过任何中间方，不被抽取任何佣金。当争议发生时，规则由代码执行，仲裁由 NFT 门控的社区成员投票，结果不可被单边撤销。

**对整个行业而言**，SuperGuild 是一次对"平台取代工会"这一 Web2 范式的反叛。历史上，工会是劳动者集体保护自身权益的组织形式；在链上，SuperGuild 通过协议和 DAO，实现了一个不需要信任任何单一实体的去中心化协作网络。

我们相信，超级个体时代已经到来。SuperGuild 是为这个时代而生的基础设施。
